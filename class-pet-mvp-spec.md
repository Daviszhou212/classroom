# 班级电子宠物管理系统（小学版）MVP 规格说明（离线 HTML）

> 本文档用于指导 Codex Agent 编程实现 **MVP（简单版本）**。  
> 只明确**框架、功能、页面、数据结构、规则与验收标准**；不提供具体编程步骤或代码实现建议，由 Codex Agent 自行决定实现细节。

---

## 1. 项目目标

在小学班级场景中提供一个**完全离线部署**的“班级电子宠物管理系统”，用于：
- 老师快速给学生发放奖励（积分/星星）。
- 学生在老师开启的班会喂养模式中，或由老师代操作，用奖励购买食物并喂养宠物。
- 用宠物的可视化状态与成长，形成持续激励。

---

## 2. 关键约束（必须满足）

### 2.1 离线与部署
- **完全离线运行**：不得依赖任何外网 API、CDN、在线字体、在线图标库。
- 交付物为一套可本地部署的静态资源（HTML/CSS/JS/图片/字体等），可通过：
  - 直接打开 `index.html` 使用；或
  - 在本地启动静态服务器后访问（仍离线）。
- 所有依赖（如有）必须**本地随项目交付**，不得运行时联网拉取。

### 2.2 隐私与数据
- 不采集不必要的个人信息；MVP 仅需：学生姓名/座位号/分组（可选）。
- 数据存储在本机浏览器本地（如 IndexedDB / LocalStorage 等），**刷新不丢**。
- 必须提供 **导出备份** 与 **导入恢复** 功能（JSON）。

### 2.3 角色与权限（MVP）
- 需明确区分三种模式：
  - **教师模式**：管理学生、发放奖励、配置、导入导出、班会喂养模式入口。
  - **班会喂养模式**：仅能由老师先开启；学生在老师监督下选择自己并执行喂养。
  - **展示模式**：课堂投影专用，只读展示，不允许喂养或其他写操作。
- 如保留“学生查看页”，应仅用于查看宠物状态与积分，不承担喂养职责。
- 教师模式入口必须有**本地口令/PIN**保护（最低限度防误触）。

---

## 3. MVP 功能范围（必须实现）

> 本节为 MVP 必做项；未列入的功能默认不做（例如“勋章/成就”不在 MVP 中）。

### 3.1 学生管理
- 添加学生（至少字段：姓名、座位号）。
- 编辑/删除学生（删除需二次确认）。
- 学生列表展示（支持按座位号排序；支持搜索/过滤可选）。

### 3.2 宠物系统（每个学生一个宠物）
- 每位学生自动拥有一个宠物档案（MVP 可固定同一宠物类型，不要求多品类）。
- 新增学生时宠物类型可随机分配；每位学生在首次有效喂养前，拥有 1 次重新领养并更换宠物类型的机会。
- 宠物需要至少包含以下可视化状态：
  - **等级 Level**（整数）
  - **经验 XP**（用于升级）
  - **饥饿 Hunger**（0–100）
  - **心情 Mood**（0–100）
- 宠物页面需要展示：宠物形象（可用静态图）、等级、状态条、可执行动作（喂食）。

### 3.3 奖励（积分/星星）系统
- 每个学生拥有一个可消费余额：**星星/积分**（统一称“积分”即可）。
- 教师模式支持：
  - 给单个学生加分（+）
  - 给单个学生扣分（-）（可选：允许关闭扣分；若实现扣分，则必须记录理由）
  - 加/扣分需要记录“理由”（可使用预设模板 + 自定义输入）
- 必须有“最近操作/流水”展示（至少显示：时间、学生、变动、理由、操作者=教师）。

### 3.4 商店与喂养（消费闭环）
- 提供一个“商店/食物列表”（MVP 固定 2–5 种食物即可）：
  - 食物名称
  - 价格（消耗积分）
  - 效果（至少影响 Hunger；可选影响 Mood/XP）
- 喂养流程要求：
  - 班会喂养模式中由学生选择食物，或由教师在学生详情中代操作 -> 扣除积分 -> 宠物状态变化
  - 若积分不足，应提示并禁止交易
- 必须有“喂养记录”在流水中可追溯（与奖励流水同一体系即可）。

### 3.5 展示模式（课堂投影友好）
- 从主页一键进入 **展示模式**（全屏友好）：
  - 支持按学生轮播/翻页展示宠物卡片
  - 显示姓名/座位号、宠物形象、等级与状态
- 展示模式不得暴露教师 PIN 与管理入口；退出后仅返回主页。
- 展示模式必须保持只读，不允许喂养、加分或其他会修改数据的操作。

### 3.6 班会喂养模式
- 只能从教师模式进入，且进入前必须完成教师 PIN 验证。
- 页面需展示按座位号排序的学生名单，学生自选自己进入。
- 单个学生进入后，可先使用 1 次重新领养机会更换宠物类型；随后可连续喂养多次，也可以本次跳过并返回名单。
- 模式内应隐藏或禁用全局管理导航，避免学生借老师已登录状态进入教师管理界面。
- 本次会话中已操作过的学生可做视觉标记，但不强制限制再次进入。
- 会话仅由老师手动结束；结束后返回教师模式。

### 3.7 数据备份与恢复
- **导出 JSON**：包含所有学生、宠物、积分余额、流水、配置。
- **导入 JSON**：支持恢复到导出时状态；必须做基本校验（例如版本号/结构存在）。
- 导入动作需要二次确认（避免误覆盖）。

---

## 4. 明确不做（MVP 非目标）

- 勋章/成就系统（本轮明确不做）
- 复杂社交：评论、点赞、私聊、互送等
- 付费、抽卡、随机箱等机制
- 多端实时同步（MVP 只保证单设备离线可用；可通过导出/导入手动迁移）

---

## 5. 页面与导航（建议信息架构）

> 允许单页应用（SPA）或多页结构；关键是功能可达、离线可用。

### 5.1 公共
- 顶部：系统标题 + 当前模式标识（教师/班会喂养/展示）
- 明确入口：返回首页、导入导出（仅教师）

### 5.2 教师模式页面
1. **教师首页 / 仪表盘**
   - 快捷入口：学生管理、发放奖励、班会喂养模式、导入导出、系统设置
2. **学生管理**
   - 列表 + 新增/编辑/删除
3. **发放奖励**
   - 选择学生 -> 输入变动值（或 +1/+2 快捷）-> 选择理由 -> 提交
4. **学生详情（教师视图）**
   - 学生积分余额、宠物状态、快速喂养、最近流水
5. **导入/导出**
   - 导出 JSON
   - 导入 JSON（覆盖式恢复）
6. **系统设置**
   - 教师 PIN 设置/修改
   - 基本规则（可选：等级阈值、每日上限等；若不实现可先固定在代码中）

### 5.3 学生展示模式页面
- **我的宠物**
  - 选择学生（方式由实现决定：列表/座位号输入）
  - 展示宠物状态 + 积分余额
  - 仅查看，不提供喂养入口

### 5.4 班会喂养模式页面
- 显示学生名单卡片，按座位号排序
- 学生选择自己后进入宠物详情、一次重新领养入口与食物列表
- 可连续喂养，也可跳过并返回名单
- 仅保留会话内控制（如返回名单、结束会话）

### 5.5 展示模式页面
- 全屏卡片轮播/翻页
- 仅展示关键数据；隐藏管理功能；不提供任何写操作

## 6. 数据模型（MVP 必须字段）

> 具体存储方式由 Codex Agent 决定；需满足离线持久化与可导入导出。

### 6.1 Student
- `id`：唯一标识
- `name`：姓名
- `seatNo`：座位号（字符串或数字）
- `group`：分组（可选）

### 6.2 Pet
- `id`
- `studentId`
- `petType`
- `petName`（可选，默认可自动生成）
- `level`（int）
- `xp`（int）
- `hunger`（0–100）
- `mood`（0–100）
- `reAdoptAvailable`（bool，是否仍有 1 次重新领养机会）
- `reAdoptedAt`（重新领养时间戳，可为空）
- `updatedAt`（最后一次状态变动时间戳）

### 6.3 Wallet / Balance（可并入 Student）
- `studentId`
- `points`（int，>=0）

### 6.4 CatalogItem（食物/商品）
- `id`
- `name`
- `pricePoints`
- `effects`（结构化：如 hunger +X、mood +Y、xp +Z）

### 6.5 LedgerEntry（流水/日志）
- `id`
- `timestamp`
- `studentId`
- `type`（建议枚举：`award`/`deduct`/`buy_food`/`feed`/`re_adopt`/`adjust`）
- `deltaPoints`
- `deltaXp`（可选）
- `deltaHunger`（可选）
- `deltaMood`（可选）
- `reason`（字符串；允许为空但建议必填）
- `operator`（固定为 `teacher`；若未来扩展再加）

### 6.6 AppConfig
- `schemaVersion`
- `teacherPinHash` 或 `teacherPin`（由实现决定；MVP 可简化，但不得明文展示在 UI）
- 其他规则（可选）

## 7. 业务规则（MVP 需要明确的“可解释性”）

> 数值可由 Codex Agent 固化为默认配置；但规则必须清晰一致。

### 7.1 积分规则
- 积分余额不得为负数。
- 扣分（若启用）不允许导致负数；应提示并阻止操作，或将余额归零（由实现决定，但需一致）。
- 所有变动必须进入流水。

### 7.2 喂养规则
- 喂养必须先校验积分足够，再扣费，再应用状态变化。
- Hunger/Mood 上限为 100，下限为 0（溢出需截断）。
- 喂养应进入流水（至少记录食物名与效果）。
- 每个学生仅有 1 次重新领养机会，且仅可在首次有效喂养前使用。
- 重新领养只更换宠物类型，不改变积分、XP、Hunger、Mood，但必须进入流水。

### 7.3 升级规则
- 以 XP 驱动 Level（可设定一个简单阈值公式或固定表）。
- 发生喂养/奖励等导致 XP 变化时，应触发升级检查。
- 升级可带来视觉反馈（可选：提示弹窗/动画）。

### 7.4 时间相关（MVP 可简化）
- MVP 不强制实现“每日衰减”；如实现，需：
  - 只依赖本地时间；
  - 在应用启动或打开宠物页时计算；
  - 避免因系统时间回拨导致异常（至少要有基本保护）。

## 8. UI/UX 基线（课堂可用性）

- 默认中文界面（简体）。
- 教室投影可读：主要按钮与文字需要足够大。
- 操作路径短：教师给某学生加 1 分，目标 <= 3 次点击/触控。
- 明确提示与二次确认：
  - 删除学生
  - 导入覆盖
  - 重置/清空数据（若提供）

---

## 9. 质量与验收标准（Definition of Done）

MVP 交付需满足以下验收点：

### 9.1 离线验收
- 断网环境下，所有页面功能正常（无外链资源加载失败导致的不可用）。
- 所有资源均来自本地项目目录。

### 9.2 数据持久化
- 刷新页面不丢数据。
- 关闭浏览器后再次打开仍可恢复。

### 9.3 功能验收
- 能新增学生并生成对应宠物档案。
- 教师能为学生加分，并在流水中看到记录。
- 教师详情页能购买/喂食，积分减少，宠物状态增加，流水可追溯。
- 班会喂养模式下，学生可在老师监督下选择自己，首次有效喂养前可重新领养 1 次，并可连续喂养或跳过返回名单。
- 展示模式可从主页进入并轮播/翻页查看所有学生宠物信息，且保持只读。
- 导出 JSON 后可在同一浏览器清空数据再导入恢复，数据一致。

### 9.4 可维护性（最低要求）
- 数据结构有明确 `schemaVersion`（便于后续迭代升级）。
- 关键规则集中配置（即使是常量，也应集中，避免散落）。
